# implement-code: 実装コードスキル

テストが通るようにコードを実装する（TDDのGreen→Refactor Phase）。

## モード判定

```
IF .dev-docs/review-report.md が存在する AND ユーザーが「レビュー指摘を修正して」等と指示
  → レビュー指摘反映モード
ELSE
  → 通常実装モード
```

---

## 通常実装モード

### Step 1: 入力の読み込み

1. `.dev-docs/spec.md` を読み込む（存在すれば）
2. `.dev-docs/plan.md` を読み込む（存在すれば — アーキテクチャ決定・設計方針を把握する）
3. `.dev-docs/tasks.md` を読み込む（存在すれば）
4. テストコードを読み込む

### Step 2: 未完了タスクの取得

`.dev-docs/tasks.md` から未完了タスク（`- [ ]`）を依存順に取得する。
`.dev-docs/tasks.md` が存在しない場合は、テストファイルから実装対象を特定する。

### Step 3: タスクごとの実装

各タスクに対して以下を実行する:

#### 3a. テストの分析

対応するテストファイルを読み、テストが期待する振る舞いを分析する。

#### 3b. 実装（Green）

- 既存コードがある場合、設計パターン・命名規則に準拠して実装する
- テストが通る最小限のコードを実装する

#### 3c. テスト実行

テストを実行して全てPassすることを確認する。

**テストがPassしない場合、以下の段階的アプローチで修正する:**
1. **直接修正**: エラーメッセージを分析し、直接的な修正を適用する
2. **アプローチ見直し**: 仕様書を再確認し、実装アプローチを根本から見直す
3. **ユーザー判断**: テスト自体の妥当性を検証し、テストが不適切な可能性も含めてユーザーに判断を仰ぐ

3段階を経てもPassしない場合はユーザーに報告し、タスクをスキップするか対応を相談する。

#### 3d. リファクタリング（Refactor）

テストがPassした後、以下の観点でリファクタリングする:
- 重複排除
- 命名改善
- 構造整理

#### 3e. テスト再実行

リファクタリング後にテストを再実行し、全てPassを維持することを確認する。

#### 3f. タスク完了の記録

`.dev-docs/tasks.md` の該当タスクのチェックボックスを `[x]` に更新する。

#### 3g. コミット

タスク単位でコミットする。

**コミット前の確認:**
1. `git status` で変更ファイルを確認する
2. タスクに関連するファイルのみがステージングされていることを確認する
3. 意図しない差分（無関係なファイルの変更等）があれば除外する

**コミット戦略:**
- ブランチの作成・切り替えは行わない（ユーザーが事前にブランチ・worktreeを準備する前提）
- コミットメッセージのフォーマット: `<prefix>: <タスク名>`
- Conventional Commits プレフィックス:
  - 新機能: `feat: <タスク名>`
  - バグ修正: `fix: <タスク名>`
  - リファクタリング: `refactor: <タスク名>`
  - テスト: `test: <タスク名>`
  - ドキュメント: `docs: <タスク名>`
- プレフィックスはタスクの性質から自動判定する

## レビュー指摘反映モード

### Step 1: レビューレポートの読み込み

`.dev-docs/review-report.md` を読み込む。

### Step 2: 指摘の抽出

`[Critical]` と `[Warning]` の指摘を抽出する。

### Step 3: 指摘ごとの修正

各指摘に対して:
1. 対象ファイル・行番号を特定する
2. 改善案に基づいて修正を適用する
3. テストを実行して全てPassすることを確認する

### Step 4: コミット

全指摘の修正後にコミットする:
```
fix: レビュー指摘の修正
```

注意: `.dev-docs/review-report.md` は上書きしない（次の `review-implements` で再生成される）。
