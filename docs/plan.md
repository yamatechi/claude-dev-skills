# 実装計画: Claude Dev Skills

## 実装方針

- 各スキルは独立した `SKILL.md` ファイルとして実装する
- プロンプトエンジニアリングが主な実装作業（コードではなくMarkdownプロンプト）
- 依存関係のないスキルから順に実装し、動作確認を行う

## 実装順序

### Phase 1: 基盤整備

プロジェクト構造の準備とスキルの共通基盤を整える。

1. **プロジェクト構造のセットアップ**
   - `skills/` ディレクトリ構造の作成
   - README.md の作成（インストール手順・使い方）

### Phase 2: コアスキル実装

ワークフローの上流から順に実装する。

2. **create-spec スキル**
   - 最初に実装する（他スキルへの入力ドキュメントを生成するため）
   - 新規作成モード・差分更新モードの両方を実装
   - タスク分解ルール・変更履歴ルールをプロンプトに組み込む

3. **create-tests スキル**
   - spec の出力（spec.md, tasks.md）を入力として使用
   - 技術スタック検出ロジックをプロンプトに組み込む
   - ファイル配置ルール・命名規則の検出をプロンプトに組み込む

4. **implement-code スキル**
   - test-gen の出力（テストコード）を入力として使用
   - TDD Green→Refactor フローをプロンプトに組み込む
   - コミット戦略・タスク完了更新をプロンプトに組み込む
   - レビュー指摘反映モードを実装

5. **review-implements スキル**
   - implement の出力（実装コード）を入力として使用
   - レビュー観点・重要度分類をプロンプトに組み込む
   - レビューレポート構造（チャット出力 + ファイル出力）を実装

6. **create-pr スキル**
   - review の出力（review-report.md）を入力として使用
   - PR本文テンプレート・ラベル導出・レビュアー設定をプロンプトに組み込む
   - gh CLI 連携をプロンプトに組み込む

### Phase 2.5: オーケストレーションスキル実装

7. **orchestrator スキル**
   - 5つの個別スキルを統合するメタスキル
   - プロジェクト状態分析・開始ポイント判定ロジックを実装
   - レビュー結果の分岐制御（テスト不足→create-tests、コード不備→implement-code）を実装
   - 自動実行中の確認ポイント制御を実装

### Phase 3: 検証

7. **動作検証**
   - Python サンプルプロジェクトで全フロー検証
   - TypeScript サンプルプロジェクトで全フロー検証
   - Go サンプルプロジェクトで全フロー検証
   - 既存プロジェクトでの動作検証
   - 独立使用の動作検証

## アーキテクチャ決定

### SKILL.md のプロンプト設計方針

- **1ファイル完結**: 各スキルは1つの `SKILL.md` に全ロジックを記述する（補助ファイルは使わない）
- **宣言的**: 手順を明示的に記述し、Claude が判断に迷わないようにする
- **フォールバック明示**: 検出できない場合・ファイルがない場合の動作を必ず記述する
- **ユーザー確認ポイント明示**: 自動で進める部分と確認が必要な部分を明確に分ける

### なぜ補助ファイルを使わないか

- スキルのポータビリティを最大化する（SKILL.md 1ファイルで配布可能）
- 依存関係を減らしてメンテナンスを容易にする
- Claude Code のスキル機能はプロンプトベースのため、ロジックは全てプロンプト内に記述する方が自然

### ツール許可の方針

全スキルで `Read Grep Glob Bash Write Edit Agent` を許可する:
- `Read/Grep/Glob`: 既存コード・設定ファイルの調査
- `Bash`: テスト実行、git操作、gh CLI操作
- `Write/Edit`: ドキュメント・コード・テストの生成・編集
- `Agent`: 複雑な調査の委譲（技術スタック検出等）
