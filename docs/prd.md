# PRD: Claude Dev Skills

## 概要

仕様駆動開発（Spec-Driven Development）とテスト駆動開発（TDD）のワークフローを支援する Claude Code スキルセット。
6つのスキル（5つの個別スキル + 1つのオーケストレーションスキル）を使うことで、仕様→テスト→実装→レビュー→PR作成の一貫した開発フローを実現する。

## 背景・課題

- 仕様が曖昧なまま実装に入ると、手戻りが発生しやすい
- テストを後回しにすると、カバレッジが不十分になりがち
- コードレビューの観点が属人化しやすい
- Claude Code で開発する際、毎回同じ指示を繰り返す手間がある
- 既存プロジェクトへの機能追加時にも一貫したフローが必要

## ゴール

- 仕様→テスト→実装→レビュー→PR作成の開発フローをスキルとして標準化する
- どのプロジェクト・言語でも汎用的に使える
- 新規プロジェクト・既存プロジェクトの両方に対応する
- 各フェーズの品質を担保する

## ワークフロー

### 推奨フロー

```
spec → test-gen → implement → review → pr
                                 ↓
                    指摘あり → implement で修正 → review（再レビュー）
                    テスト不足 → test-gen で追加 → implement → review
                                                      ↓
                                                承認 → pr
```

### 独立使用時の前提条件

各スキルは独立して使用可能だが、以下の前提条件を満たす必要がある:

| スキル | 前提条件 |
|--------|---------|
| `create-spec` | なし（最初に実行可能） |
| `create-tests` | 仕様書（`docs/spec.md`）が存在するか、ユーザーが口頭で仕様を伝えること |
| `implement-code` | テストコードが存在すること。仕様書があれば参照する |
| `review-implements` | 実装コードが存在すること。仕様書・テストがあれば参照する |
| `create-pr` | ブランチにコミットが存在すること。`review-implements` が承認済みであること（推奨） |

### 仕様変更時の差分更新フロー

仕様変更が発生した場合:
1. `create-spec` で仕様書を差分更新（変更箇所を明示）
2. `create-tests` で影響を受けるテストを更新・追加
3. `implement-code` で影響を受ける実装を更新
4. `review-implements` で変更全体をレビュー
5. `create-pr` で変更内容を反映したPRを作成（または既存PRの本文を更新）

## スキル構成

### 1. `create-spec` — ドキュメント作成スキル

**目的**: ユーザーの要望からPRD・仕様書・実装計画・タスク一覧を作成する

**入力**: ユーザーの要望（自然言語）

**出力**:
- `docs/prd.md` — プロダクト要件定義書
- `docs/spec.md` — 技術仕様書
- `docs/plan.md` — 実装計画
- `docs/tasks.md` — タスク一覧

**動作（新規作成モード）**:
1. ユーザーの要望をヒアリング・整理
2. 既存プロジェクトの場合、既存コード・設計パターンを調査
3. 不明点があれば質問
4. 各ドキュメントを順次生成
5. ユーザーに確認・承認を得る

**動作（差分更新モード）**:
1. 既存ドキュメント（prd.md, spec.md, plan.md, tasks.md）を読み込む
2. ユーザーの変更要望をヒアリング
3. 影響範囲を分析し、変更が必要なドキュメントを特定
4. 各ドキュメントの変更箇所を更新（変更履歴セクションに変更理由・日付を記録）
5. ユーザーに差分を提示し、確認・承認を得る

**既存プロジェクト対応**:
- 既存コードの構造・設計パターンを調査し、仕様書に反映する
- 既存テストのスタイル・構成を把握し、タスク一覧に反映する
- 既存コードへの影響範囲を実装計画に明記する

### 2. `create-tests` — テストコード作成スキル

**目的**: 仕様書からテストコードを生成する（TDDのRed Phase）

**入力**: 仕様書（`docs/spec.md`）またはユーザー指定のドキュメント

**出力**: テストコード

**動作**:
1. 仕様書を読み込む
2. プロジェクトの技術スタック・テストフレームワークを自動検出（→「技術スタック検出戦略」参照）
3. 既存テストがある場合、そのスタイル・構成に合わせる
4. テストケースを設計（正常系・異常系・エッジケース）
5. テストコードを生成（出力先は「ファイル配置ルール」に従う）
6. テストを実行して全てFailすることを確認（Red）

**異常系フロー**:
- テストが最初からPassする場合: 実装が既に存在する可能性を警告し、テストの妥当性を再検討する
- テスト実行自体が失敗する場合（構文エラー等）: 修正してから再実行する

**責務**: テストの作成・追加は常にこのスキルが担う（`review-implements` でカバレッジ不足が指摘された場合も `create-tests` で対応）

### 3. `implement-code` — 実装コードスキル

**目的**: テストが通るようにコードを実装する（TDDのGreen→Refactor Phase）

**入力**: 仕様書 + テストコード

**出力**: 実装コード

**動作**:
1. 仕様書とテストコードを読み込む
2. 既存コードがある場合、設計パターンに準拠して実装する
3. テストが通る最小限のコードを実装（Green）
4. テストを実行して全てPassすることを確認
5. リファクタリング（Refactor）
6. テストを再実行してPassを維持
7. `docs/tasks.md` の該当タスクのチェックボックスを完了に更新する
8. タスク単位でコミット

**コミット戦略**:
- スキルはブランチの作成・切り替えを行わない（ユーザーが事前にブランチ・worktreeを準備する前提）
- コミットメッセージにはConventional Commitsのプレフィックス（feat/fix/refactor等）を付与する
- コミットメッセージのフォーマット: `<prefix>: <タスク名>`

**レビュー指摘反映モード**:
- `review-implements` のレビューレポートを入力として受け取り、`[Critical]` `[Warning]` の指摘に対して実装を修正する
- 修正後にテストを再実行してPassを維持

**異常系フロー**:
- テストがPassしない場合: 以下の段階的アプローチで修正を試行する
  1. エラーメッセージを分析し、直接的な修正を適用
  2. 仕様書を再確認し、実装アプローチを見直す
  3. テスト自体の妥当性を検証し、必要に応じてユーザーに判断を仰ぐ
  - 3段階を経てもPassしない場合はユーザーに報告する

### 4. `review-implements` — 実装レビュースキル

**目的**: 実装コードの品質・仕様適合性をレビューする

**入力**: 仕様書 + テストコード + 実装コード

**出力**:
- チャット上にマークダウン形式でレビューレポートを表示
- `docs/review-report.md` にレビューレポートを保存（`create-pr` スキル等が参照するため）

**レビューレポート構造**:

```markdown
## レビューレポート

### 総合判定: 承認 / 要修正 / 要再設計

### 指摘一覧

#### [Critical] 指摘タイトル
- **対象**: ファイルパス:行番号
- **内容**: 問題の説明
- **改善案**: 具体的な修正提案

#### [Warning] 指摘タイトル
- **対象**: ファイルパス:行番号
- **内容**: 問題の説明
- **改善案**: 具体的な修正提案

#### [Info] 指摘タイトル
- **対象**: ファイルパス:行番号
- **内容**: 問題の説明
- **改善案**: 具体的な修正提案

### 次のアクション
- 推奨するスキルと対応内容
```

**動作**:
1. 仕様書・テスト・実装コードを読み込む
2. 以下の観点でレビュー:
   - 仕様との適合性
   - テストカバレッジの十分性
   - コード品質（可読性・保守性）
   - セキュリティ
   - パフォーマンス
   - 既存コードとの一貫性（命名規則・設計パターン・ディレクトリ構成の準拠）
3. 問題点と改善提案をレポート構造に従って出力
4. 重要度（Critical / Warning / Info）で分類
5. 次のアクションを提案:
   - Critical/Warning あり → `implement-code` で修正を推奨
   - テストカバレッジ不足 → `create-tests` でテスト追加を推奨
   - 問題なし → 承認

**異常系フロー**:
- 仕様書が存在しない場合: 実装コードのみでレビュー可能だが、仕様適合性の観点はスキップする旨を明示する

### 5. `create-pr` — Pull Request作成スキル

**目的**: 実装内容からPull Requestを作成する

**入力**: 仕様書 + タスク一覧 + レビューレポート + gitの差分

**出力**: GitHub Pull Request

**動作**:
1. `review-implements` の総合判定を確認し、承認済みかチェックする
   - 承認済みでない場合: 警告を表示し、続行するかユーザーに確認する
2. 仕様書（`docs/prd.md`, `docs/spec.md`）・タスク一覧（`docs/tasks.md`）を読み込む
3. `git log` と `git diff` でブランチの変更内容を把握する
4. PRタイトルを生成（70文字以内、Conventional Commitsプレフィックス付き）
5. PR本文を生成（→「PR本文テンプレート」参照）
6. ラベルを自動設定（コミットメッセージのConventional Commitsプレフィックスから導出: feat→feature, fix→bugfix, refactor→refactor 等。プロジェクトの既存ラベルに存在するもののみ付与）
7. レビュアーを設定（ユーザーに確認、または `.github/CODEOWNERS` から推定）
8. `gh pr create` でPRを作成
9. 作成したPRのURLをユーザーに返す

**PR本文テンプレート**:

```markdown
## Summary
<!-- 仕様書から要約した変更の概要（1-3箇条書き） -->

## Changes
<!-- タスク一覧から生成した変更内容の詳細 -->

## Test plan
<!-- テストの実行方法と確認観点 -->

## Review notes
<!-- レビューレポートの総合判定と残存する Info 指摘があれば記載 -->
```

**カスタマイズ**:
- プロジェクトに `.github/PULL_REQUEST_TEMPLATE.md` が存在する場合、そのテンプレートに従う
- ラベル候補はプロジェクトの既存ラベルから選択する（存在しないラベルは付与しない）
- レビュアーは `.github/CODEOWNERS` → ユーザー指定 → 手動確認の優先順で決定する

**異常系フロー**:
- リモートにブランチがpushされていない場合: pushするかユーザーに確認する
- PRが既に存在する場合: 既存PRの本文を更新するかユーザーに確認する

## タスク分解ルール

### タスクのスキーマ

各タスクには以下の情報を含める:

```markdown
- [ ] タスク名（テストファイルパス）
  - 依存: 依存タスク名（あれば）
```

### 粒度ルール

- **1タスク = 1つの論理的な機能単位**
- 基本的に1テストファイルに対応するが、テストファイルが大きくなる場合はテストスイート内のグループ（describe/context単位）で分割する
- 1タスクが1コミットに対応する

### 粒度例

```markdown
## 実装タスク

- [ ] ユーザーモデルのバリデーション（tests/models/user_validation_test.rb）
- [ ] ユーザーモデルのCRUD操作（tests/models/user_crud_test.rb）
  - 依存: ユーザーモデルのバリデーション
- [ ] ログインAPIの実装（tests/api/login_test.rb）
  - 依存: ユーザーモデルのCRUD操作
- [ ] 認証ミドルウェアの実装（tests/middleware/auth_test.rb）
  - 依存: ログインAPIの実装

## セットアップタスク

- [ ] プロジェクト初期設定（DB接続・マイグレーション等）
- [ ] テスト環境の構築
```

## 技術スタック検出戦略

スキルはプロジェクトの技術スタックを以下の優先順で検出する:

1. **設定ファイルから検出**:
   - `package.json` → Node.js / npm（テスト: jest, vitest, mocha 等を `devDependencies` から特定）
   - `pyproject.toml` / `setup.py` / `requirements.txt` → Python（テスト: pytest, unittest 等）
   - `Gemfile` → Ruby（テスト: rspec, minitest 等）
   - `go.mod` → Go（テスト: 標準 testing パッケージ）
   - `Cargo.toml` → Rust（テスト: 標準テスト機能）
   - `build.gradle` / `pom.xml` → Java/Kotlin（テスト: JUnit, Kotest 等）
   - `composer.json` → PHP（テスト: PHPUnit 等）
2. **既存テストファイルから検出**: テストディレクトリ・ファイルの命名パターンを分析
3. **フォールバック**: 検出できない場合はユーザーに質問する

### テスト実行コマンドの検出

- `package.json` の `scripts.test` → `npm test`
- `Makefile` の `test` ターゲット → `make test`
- 検出できない場合は言語のデフォルト（`pytest`, `go test ./...`, `rspec` 等）を使用
- それでも不明な場合はユーザーに確認する

## ファイル配置ルール

テスト・実装コードの出力先は以下の戦略で決定する:

1. **既存プロジェクトの構造に従う**: 既存のテスト・ソースディレクトリを検出し、そのパターンに合わせる
2. **既存ファイルがない場合**: 言語の標準的な構成を提案し、ユーザーに確認する
3. **テストファイルの命名規則**: 既存テストの命名パターン（`*_test.go`, `*.test.ts`, `*_spec.rb` 等）に合わせる

## ファイル配置構造

```
skills/
├── orchestrator/
│   └── SKILL.md
├── create-spec/
│   └── SKILL.md
├── create-tests/
│   └── SKILL.md
├── implement-code/
│   └── SKILL.md
├── review-implements/
│   └── SKILL.md
└── create-pr/
    └── SKILL.md
```

## スキル間のデータ連携

スキル間のデータ受け渡しはファイルシステム経由で行う。各スキルは所定のパスのファイルを読み書きすることで連携する:

- `create-spec` → `docs/` 配下のドキュメント → `create-tests` が読む
- `create-tests` → テストコード → `implement-code` が読む
- `implement-code` → 実装コード → `review-implements` が読む
- `review-implements` → レビューレポート（チャット出力 + `docs/review-report.md`） → ユーザーが `implement-code` または `create-tests` に指示
- `review-implements`（承認） → `create-pr` が仕様書・タスク一覧・`docs/review-report.md`・git差分を読んでPR作成
- `orchestrator` → 上記フロー全体を自動制御。プロジェクト状態を分析し開始ポイントを判定、各スキルを順次実行する

## 技術要件

- Claude Code のスキル機能（`SKILL.md`）を利用
- プロジェクト・言語に依存しない汎用設計
- 技術スタック・テストフレームワークは動的に検出（→「技術スタック検出戦略」参照）
- 各スキルは独立して使用可能（→「独立使用時の前提条件」参照）
- `create-pr` スキルは `gh` CLI（GitHub CLI）がインストール・認証済みであることを前提とする

## スコープ外

- CI/CD連携
- 特定言語・フレームワーク固有の最適化
- デプロイ関連の機能

## 成功指標

最低3つの異なる言語（Python / TypeScript / Go 等）のサンプルプロジェクトで、以下を全て満たすこと:

1. `create-spec` → ドキュメント4点（prd.md, spec.md, plan.md, tasks.md）が生成される
2. `create-tests` → テストが生成され、全てFailする（Red）
3. `implement-code` → テストが全てPassする（Green）
4. `review-implements` → レビューレポートがレポート構造に従って出力される
5. `create-pr` → PRが作成され、タイトル・本文・ラベル・レビュアーが適切に設定される
6. 仕様変更フロー → 差分更新モードで各ドキュメント・テスト・実装が更新される

加えて:
- `orchestrator` → 全フローを自動実行し、レビュー指摘の分岐（テスト不足→create-tests、コード不備→implement-code）が正しく動作すること
- 新規プロジェクト・既存プロジェクトの両方で上記フローが動作すること
- 各スキルが独立使用時の前提条件を満たせば単独で動作すること
